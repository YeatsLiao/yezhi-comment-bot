# Hermes Agent的闭环学习架构：AI怎么把经验变成可复用的技能

大多数AI助手有个通病：每次对话结束，之前踩过的坑、跑通的流程全部清零。下次遇到同样的问题，还得从头再来一遍。

Nous Research在2026年2月发布的开源Agent框架Hermes，直接把这个问题当核心来解。上线四周GitHub Star破万，到3月底达到17400星，成为Nous Research历史上采用率最高的开源项目。三个月后，这个数字到了14万。

它做对了一件别的事——**不是记住发生了什么，而是记住什么管用**。

## 闭环学习：从经验到技能的自动转化

Hermes的核心创新是一条闭环学习链路。传统的Agent记忆系统解决的是"记住对话历史"，Hermes解决的是"把跑通的工作流自动提取成可复用技能"。

整条链路有四个环节联动：

```
任务完成 → 自主判断是否值得存技能 → 写入SKILL.md
    ↓
下次类似任务 → 向量检索召回技能 → 从已有脚手架开始而非零起步
    ↓
Honcho观察用户 → 推导偏好结论 → 后续会话预个性化
    ↓
FTS5索引全部交互 → 跨会话时序检索 → "上周二我做了什么"
```

关键在第一步：什么时候触发技能生成？Hermes设了四个明确条件：

- 调用了5次及以上工具
- 从错误中成功恢复
- 收到用户修正指导
- 走通了一套非直觉的有效流程

满足任一条件，Agent就在 `~/.hermes/skills/` 下生成一个技能文件。这个文件不是日志，是一套下次能直接照搬的操作指令集。

Nous Research内部hackathon的实测数据显示：**技能辅助任务相比冷启动执行，Token消耗降低30-85%**。另一份实践报告发现，三个自动生成的技能文档让同类研究任务完成速度提升约40%。

## 五层记忆架构：各管各的，不搅成一锅

大多数Agent把所有记忆混在一起，结果越用越臃肿。Hermes直接拆成五层，每层职责明确、存储位置固定、读取时机清晰。

```
┌──────────────────────────────────────────────────┐
│ Layer 5: FTS5全文本检索                          │
│ SQLite + 全文索引，LLM摘要后注入，跨会话时序记忆  │
├──────────────────────────────────────────────────┤
│ Layer 4: Honcho辩证用户建模                      │
│ 异步推理，存储结论而非原文，200ms检索延迟        │
├──────────────────────────────────────────────────┤
│ Layer 3: 技能记忆（过程记忆）                    │
│ SKILL.md文件，渐进式披露，只按需加载完整内容     │
├──────────────────────────────────────────────────┤
│ Layer 2: 持久记忆文件                            │
│ MEMORY.md(~2200字符) + USER.md(~1375字符)       │
├──────────────────────────────────────────────────┤
│ Layer 1: 短期推理记忆                            │
│ Transformer上下文，会话内有效，重启即清零        │
└──────────────────────────────────────────────────┘
```

### 第一层和第二层：常驻记忆

MEMORY.md和USER.md是常驻层，每次会话开始就自动注入系统提示，不需要Agent主动调用。两个文件加起来总字符上限3575，故意设得很紧凑——逼你精选，而不是无脑堆。

一个关键设计细节：会话中对这两个文件的修改，**到下次会话才生效**，不实时热更新。这避免了运行中频繁改提示词导致的缓存击穿。

### 第三层：技能记忆的渐进式披露

技能以独立Markdown文件存储在 `~/.hermes/skills/`，遵循agentskills.io开放标准。加载策略用**渐进式披露**（progressive disclosure）：默认只加载技能名和摘要，完整内容只在Agent判断当前任务需要时才载入。

这意味着不管你有多少技能，Token占用基本持平。200个技能的Agent和40个技能的上下文开销差不多，因为详细内容只在真要用时才进场。

技能不是写死不变的。Agent通过 `skill_manage` 工具支持六种操作：创建、补丁、编辑、删除、写文件、删文件。默认优先用patch方式更新——只传旧文本和替换内容，不用整个重写。这么做既保证正确性又省Token，全量重写容易把好用的部分改崩。

### 第四层：Honcho的辩证用户建模

前三层都需要Agent主动写入，Honcho不一样。它跨会话默默给你画像，不存储原始对话，而是存储**推导出的结论**。

打个比方：普通记忆存的是"用户在第47条消息里说他喜欢TypeScript"，Honcho存的是"用户偏好TypeScript"。结论而非原文，这就是它的核心设计。

Honcho用双对等架构（Dual-Peer），同时建模用户和Agent两个实体。四个工具暴露给Agent运行时调用：

| 工具 | 功能 | 延迟 |
|:---|:---|:---|
| honcho_profile | 快速获取用户画像卡片，不调LLM | 极低 |
| honcho_search | 语义搜索记忆，返回原始摘录 | 中 |
| honcho_context | 辩证问答，由Honcho的LLM综合历史生成答案 | 较高 |
| honcho_conclude | 用户表达偏好或修正时写入持久事实 | 写入操作 |

用户和Agent的画像都在会话开始时注入系统提示，检索延迟约200毫秒。

### 第五层：FTS5会话检索

所有会话存入SQLite并用FTS5建立索引。Agent检索历史时不把整段旧对话塞进上下文，而是先检索再通过LLM做摘要，只注入与当前任务相关的内容。

这一层负责**情景记忆**（记录发生了什么），技能层负责**过程记忆**（记录该怎么做）。两层分开存储，不搅在一起。

## 执行基础设施：六种后端 + 统一网关

### 六种终端后端

Hermes把Agent运行时和执行环境分离，通过 `BaseEnvironment` 接口提供六种实现：

| 后端 | 适用场景 | 特性 |
|:---|:---|:---|
| Local | 开发、个人使用 | 直接系统执行，无隔离 |
| Docker | 生产、安全敏感 | 只读根文件系统、降权、命名空间隔离 |
| SSH | 远程服务器 | 跨会话保持环境状态 |
| Daytona | 云开发 | 无服务器开发环境 |
| Singularity | HPC、科研集群 | 容器编排，计算密集型 |
| Modal | 无服务器生产 | 闲置休眠、按需唤醒，接近零空闲成本 |

切换后端只需改 `~/.hermes/config.yaml` 一行配置，Agent代码不变。

Docker模式下默认启用只读根文件系统、移除Linux特权权限、提供命名空间隔离。这些是架构级默认设置，不是可选开关。零遥测也是天生属性，不是隐私开关。

### 多平台网关

网关是一个常驻后台服务，让Hermes在所有绑定平台保持在线。支持Telegram、Discord、Slack、WhatsApp、Signal、邮件（IMAP/SMTP）、飞书/Lark、企业微信。

会话绑的是ID不是平台——在Telegram开头的对话，去终端接着聊完全没问题。Telegram还支持项目对话，在单个私聊里用话题功能跑独立工作流，每个话题自带专属技能绑定和上下文。

网关和学习循环是打通的：进来一条消息能触发技能生成，定时自动化的结果也走同一层网关回传。

## MCP双向集成

Hermes的MCP实现是双向的：

**客户端模式**——启动时连接任何已配置的MCP服务器，自动发现工具并注册为原生工具。断线重连用指数退避（1s→2s→4s→8s→16s，最多5次）。

**服务器模式**——v0.6.0新增，通过 `hermes mcp serve` 把Hermes的对话和会话暴露给任何MCP兼容客户端（Claude Desktop、Cursor、VS Code）。客户端可以浏览对话、读取消息、跨会话搜索。

这意味着Hermes可以作为其他Agent的持久记忆后端——它不只是个Agent，还能给别的Agent当大脑。

## 效果数据与工程取舍

| 指标 | 数据 | 来源 |
|:---|:---|:---|
| 技能辅助任务Token降低 | 30-85% | Nous Research hackathon |
| 研究任务完成速度提升 | ~40% | 实践报告 |
| Honcho检索延迟 | ~200ms | Plastic Labs |
| MEMORY.md+USER.md总上限 | 3575字符 | 架构设计 |
| 技能默认加载内容 | 仅名称+摘要 | 渐进式披露 |
| 支持平台数 | 8+ | v0.6.0 |

但也要说清楚Hermes的局限。

技能本质上是**精密的提示词注入，不是真正的学习**。Agent并没有更新自己的权重，它只是在外部文件系统里存了一套操作手册，下次遇到类似任务时先读手册再干活。这跟人类积累经验有本质区别——人改的是神经元连接，Hermes改的是文件系统。

Honcho的用户建模缺乏公开的A/B测试基准。没有遗忘/衰减机制意味着技能库只增不减，长期使用后可能产生过时技能干扰。文档覆盖也有缺口，企业级特性（SSO、审计日志、RBAC）尚未提供。

## Atropos飞轮：从Agent使用到模型训练

Hermes最值得关注的长期方向是Atropos——Nous Research的开源分布式RL训练框架。Atropos在Hermes 4训练中驱动约1000个任务特定验证器做拒绝采样，筛选高质量推理轨迹。

同一个Atropos集成出现在Agent运行时中，用于RL训练和轨迹捕获。这意味着**Agent的使用数据可以直接回流为模型训练数据**——用户用得越多，积累的技能和轨迹越多，下一版Hermes模型就越强。

这是一条从Agent使用→训练数据→更好模型的飞轮，也是Hermes区别于纯工程框架的底层护城河。

对于想上生产的团队，务实的选型建议是：如果任务高度重复且流程可固化，Hermes的技能系统ROI最高；如果任务每次都不同，技能积累的价值有限，Honcho的个性化反倒是更实用的那一层。
