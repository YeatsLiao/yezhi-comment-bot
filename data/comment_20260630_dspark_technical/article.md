# DeepSeek DSpark投机解码的工程实现：半自回归草稿+置信度动态调度

![封面](cover.jpg)

LLM推理的瓶颈不在训练，而在推理。这是2026年大模型行业的共识。

当模型参数规模从千亿膨胀到万亿，训练成本可以用钱堆，但推理成本是持续性的——每服务一个用户、每生成一个token，都要消耗算力。DeepSeek的解法不是换更大的模型，而是给现有模型换一套"变速箱"：在V4-Pro和V4-Flash基座上附加一个推测解码模块，per-user生成速度提升60%-85%。

这套模块叫DSpark，全称Confidence-Scheduled Speculative Decoding with Semi-Autoregressive Generation。论文由DeepSeek-AI联合北京大学发布，梁文锋署名。

```
传统逐个生成（自回归）              推测解码（Speculative Decoding）
├─ 大模型前向 × N步                  ├─ 小模型草稿 × 1步（生成γ个token）
├─ 每步延迟 ≈ 大模型推理时间           ├─ 大模型验证 × 1步（并行验证γ个token）
└─ 总延迟 = N × T_target             ├─ 接受一致的 → 保留
                                      ├─ 拒绝的位置 → 大模型重新生成
                                      └─ 总延迟 ≈ (N/γ) × T_target + N × T_draft
```

## 问题定义：推测解码的两大痛点

推测解码（Speculative Decoding）的基本思路并不复杂：用小模型（draft model）快速生成候选token序列，再用大模型（target model）一次性验证整段——接受一致的，拒绝不一致的，然后从第一个拒绝的位置重新生成。

这个方案的理论收益很清晰：小模型推理快但质量差，大模型推理慢但质量高。如果小模型猜对的token足够多，大模型验证的步数就远小于逐个生成的步数，整体吞吐量自然提升。

但在工程落地中，现有方案踩了两个坑：

**第一，草稿质量衰减。** 纯并行生成的小模型（如DFlash）一次前向输出整块候选token，token之间缺乏序列依赖。结果是前缀猜得准，后缀崩得快——行业叫"suffix decay"。一块32个token的草稿，前8个接受率可能90%，后8个直接跌到40%。

**第二，验证长度僵化。** 传统方案用固定长度γ做验证（比如每次验4个或8个token）。低并发时，γ太小浪费了大模型的并行验证能力；高并发时，γ太大导致大量无效验证，GPU算力被草稿模型和大模型两头吃掉。

DSpark的设计目标很明确：既要并行高吞吐，又要串行高精度；既要让草稿猜得准，又要让验证不浪费。

## 整体架构：两层解耦的设计

DSpark不是又一个新的draft model，而是一个统一框架，把"生成"和"验证"两个环节彻底解耦：

```
┌─────────────────────────────────────────┐
│  半自回归草稿生成层 (Draft Generation)    │
│  ├── 并行Backbone：一次性产出隐藏状态      │
│  └── 轻量串行模块：逐token注入前缀依赖      │
└─────────────────┬───────────────────────┘
                  │ 候选token序列 + 置信度分数
                  ▼
┌─────────────────────────────────────────┐
│  置信度动态调度验证层 (Verification)       │
│  ├── 置信度预测头：评估每个token存活概率    │
│  └── 硬件感知调度器：动态决定最优验证长度   │
└─────────────────┬───────────────────────┘
                  │ 接受/拒绝决策
                  ▼
           Target Model验证
```

两层之间通过"候选token序列 + 每个位置的置信度分数"传递信息。验证层根据置信度分布和当前系统负载，动态决定验证多少token、优先验证哪些位置。

## 核心模块一：半自回归草稿生成

DSpark的第一项创新是把草稿生成从"全并行"改成"半自回归"。

**并行Backbone**负责吞吐。它和DFlash一样，一次前向传播生成整块候选位置的基础logits和隐藏状态。这一步的延迟只取决于模型前向一次的耗时，与序列长度无关。

**轻量串行模块**负责质量。在并行Backbone产出的隐藏状态之上，叠加一个两层的轻量Transformer模块，逐token注入前缀依赖信息。具体来说，位置i的token生成时，不仅看到并行Backbone的输出，还看到位置0到i-1的已生成结果。这样，后面的token不再是"盲猜"，而是基于前面已确认的内容做条件生成。

这个设计的 trade-off 很清晰：串行模块增加了延迟，但只增加了两层Transformer的延迟，而整块草稿的接受率显著提升。论文数据显示，仅两层结构的DSpark，综合表现优于五层网络的DFlash——说明结构效率比堆参数更重要。

**抑制suffix decay的关键**在于，串行模块让token之间的条件依赖从"无"变成了"有"。前缀的上下文信息通过KV Cache逐层传递，后面token的分布不再是独立采样，而是条件采样。

## 核心模块二：置信度动态调度验证

草稿生成解决了"猜得准"的问题，验证层要解决"验得省"的问题。

DSpark在草稿模型的输出层上加了一个**置信度预测头**（Confidence Head）。每生成一个候选token，这个头就输出一个标量——表示该token被大模型接收的概率。这个头的参数量极小，推理延迟可以忽略。

拿到置信度分布后，**硬件感知前缀调度器**（Hardware-Aware Prefix Scheduler）做两件事：

**第一，动态决定验证长度。** 如果一块草稿的置信度整体偏高（比如前16个token置信度都在0.8以上），调度器会把验证长度拉满，让大模型一次多验几个。如果置信度分布参差不齐，调度器会截断低置信度后缀，避免大模型浪费时间验证"明显会错"的token。

**第二，优先验证高置信度前缀。** 调度器不是死板地从左到右验证，而是根据置信度排序，先验高置信度的token。这样做的好处是：即使验证到某个位置被拒绝，前面高置信度的token已经被接受，没有浪费。

**时序温度缩放**（Temporal Temperature Scaling）是一个容易被忽略的细节。置信度预测头在训练时可能存在系统性偏差——对早期token过度自信，对后期token信心不足。DSpark通过温度参数动态校准这种偏差，让置信度分数更接近真实的接受概率。

这个设计的工程价值在高并发场景下体现得最充分。几十上百个用户同时请求时，系统负载波动大，固定验证长度会造成严重的算力浪费。动态调度让验证资源跟着草稿质量走，好草稿多验，差草稿少验。

```
验证调度决策流程（单步）
├─ 输入：候选token序列 [t1, t2, ..., tγ] + 置信度分数 [c1, c2, ..., cγ]
├─ Step 1：温度缩放校准
│   └─ ci' = ci / τ(i)    // τ(i) 随位置i递增，抑制后期过度自信
├─ Step 2：动态截断
│   └─ 找到第一个 ci' < θ_threshold 的位置k
│   └─ 有效验证长度 γ' = min(k, γ_max)
├─ Step 3：置信度排序验证
│   └─ 对 [t1, ..., tγ'] 按 ci' 降序排列
│   └─ 大模型优先验证高置信度token
└─ 输出：接受前缀长度 + 拒绝位置（如全部接受则进入下一轮）
```

## 生产环境数据与DeepSpec开源

DSpark在DeepSeek-V4生产环境中的测试数据：

| 模型 | 基线 | per-user速度提升 |
|:---|:---|:---|
| DeepSeek-V4-Flash | MTP-1 | 60%-85% |
| DeepSeek-V4-Pro | MTP-1 | 57%-78% |

提升幅度不是实验室跑分，是生产环境真实用户请求的统计结果。V4-Flash的提升区间比V4-Pro更宽，因为轻量模型的草稿生成延迟更低，动态调度的收益空间更大。

更值得关注的不是DSpark本身，而是同步开源的**DeepSpec**代码库。MIT协议，全栈覆盖：

- **数据准备工具**：从目标模型提取训练数据，构建draft model的训练集
- **草稿模型训练**：支持DSpark、DFlash、Eagle3三种架构的端到端训练
- **推理引擎集成**：与vLLM、TGI等主流推理引擎的对接代码
- **评估脚本**：接受率、速度提升、吞吐量的标准化评测

DeepSpec的真正价值在于**通用性**。DSpark加速的是DeepSeek V4，但DeepSpec让你可以拿自己的模型——不管是Llama、Qwen还是自研模型——训练一个专属的投机解码草稿模型。

## 为什么这件事比发一个新模型更重要

DSpark的发布传递了一个信号：**大模型行业的竞争焦点正在从"参数规模"转向"推理效率"**。

训练万亿参数模型需要几亿美元，但只有少数公司玩得起。优化推理效率却是每家部署大模型的企业都面临的刚需——同样的GPU集群，推理速度快一倍，就意味着服务用户多一倍，或者成本降一半。

从工程角度看，DSpark的价值还在于**可组合性**。它不改动V4-Pro和V4-Flash的基座权重，只是一个外挂模块。企业可以在不重新训练大模型的前提下，通过附加DSpark获得60%以上的推理加速。这种"即插即用"的优化方案，比推一个新版本模型的落地门槛低得多。

当然，DSpark也有局限。它的加速效果取决于草稿模型与目标模型的"默契程度"——两者在token分布上的重叠度越高，接受率越高。如果目标模型经过大量后训练或RLHF，草稿模型可能需要同步更新才能保持最佳效果。

但无论如何，DeepSeek用DSpark证明了一件事：**在推理侧，还有大量工程优化空间等待被挖掘。**
