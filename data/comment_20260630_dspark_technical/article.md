# DeepSeek DSpark：给大模型换了一套变速箱，推理速度直接翻倍

![封面](cover.jpg)

如果你用过ChatGPT，一定遇到过这种情况：输入一个问题，模型要思考好几秒才开始回答。这几秒的延迟，不是因为模型在思考，而是因为模型在**一个字一个字地算**。

大语言模型（LLM）生成文字的方式，和人类写字完全不同。人类可以一口气写下一整句话，但LLM只能逐个token（大约等于半个汉字或一个英文单词）地生成——每生成一个字，都要把前面所有字重新过一遍神经网络。这种逐字生成的模式，就像工厂里流水线上每次只组装一个零件，效率天然受限。

DeepSeek最新的解法，不是换一条更快的流水线，而是给流水线加了一个**草稿车间**：先用一个极快的小模型把一整段话猜出来，再用大模型一次性检查对错。猜对的保留，猜错的修正。这个方案叫**DSpark**，生产环境实测per-user生成速度提升**60%-85%**。

论文由DeepSeek-AI联合北京大学发布，梁文锋署名。

```
传统逐字生成                       DSpark推测解码
├─ 大模型：今                     ├─ 小模型快速草稿：今天天气不错
├─ 大模型：天                     ├─ 大模型一次验证整句
├─ 大模型：气                     ├─ 今天天气 → 猜对了，保留
├─ 大模型：不                     ├─ 不错     → 猜错了，大模型重写
└─ 重复N次                          └─ 总延迟 ≈ 原来的 1/5 ~ 1/2
```

## 现有方案踩了两个坑

先猜后验的思路不新鲜，行业内叫**推测解码**（Speculative Decoding）。但之前的工程实现，普遍卡在两个问题上。

**第一个问题：猜着猜着就偏了。**

小模型一次生成32个字的草稿，前面几个字猜得准，越往后越离谱。原因很简单：小模型在猜第20个字的时候，并不知道前19个字长什么样——它是同时猜32个字的，字和字之间没有前后依赖。这就好比让你同时猜32个灯谜，每个灯谜的线索不能互相参考，后面的自然就靠蒙。

行业给这个现象起了个名字叫**suffix decay**（后缀衰减）：一块32个token的草稿，前8个接受率90%，后8个直接跌到40%。后半段几乎全错，大模型验证时全部推翻重写，小模型白忙活了。

**第二个问题：验证长度太死板。**

传统方案不管草稿质量好坏，每次都验固定长度（比如固定验8个字）。质量好的草稿，验8个太少，浪费了大模型的并行验证能力；质量差的草稿，验8个太多，大量无效验证把GPU算力两头吃掉。

DSpark的设计目标很明确：**让草稿越猜越准，让验证不多不少。**

## DSpark的两层架构

DSpark不是一个小模型，而是一个**统一框架**，把猜和验两个环节彻底分开设计：

```
┌─────────────────────────────────────────┐
│  第一层：半自回归草稿生成（负责猜得准）   │
│  ├── 并行Backbone：一次性草拟全文框架     │
│  └── 轻量串行模块：逐字检查，让前后文连贯  │
└─────────────────┬───────────────────────┘
                  │ 草稿 + 每个字的信心分
                  ▼
┌─────────────────────────────────────────┐
│  第二层：置信度动态验证（负责验得省）     │
│  ├── 打分器：评估每个字被大模型认可的概率  │
│  └── 调度器：动态决定验多少、先验哪些     │
└─────────────────┬───────────────────────┘
                  │ 接受/拒绝决策
                  ▼
              大模型最终确认
```

两层之间传递的信息，不只是一段草稿，还有**每个位置的信心分**。验证层根据这些分数和当前系统负载，决定验多少字、先验哪些字。

## 第一层：从全盲猜到半开卷

DSpark的第一项创新，是让小模型从同时盲猜32个字变成先看框架、再逐字确认。

**并行Backbone**负责速度。它像考试时的快速扫题——一次性浏览全文，产出每个位置的基础判断。这一步的延迟只取决于模型算一次要多久，和字数无关。

**轻量串行模块**负责质量。它在扫题的基础上，逐字检查上下文是否连贯。具体来说，猜第i个字的时候，不仅看扫题的结果，还要看前面i-1个字已经猜成了什么样。后面的字不再是盲猜，而是基于前面已确认的内容做**条件生成**。

这个设计的 trade-off 很清晰：串行检查增加了一点延迟，但只增加了两层神经网络的计算量，而整块草稿的准确率显著提升。论文数据显示，**仅两层结构的DSpark，综合表现优于五层网络的传统方案**——结构效率比堆参数更重要。

**关键改进在这里**：串行模块让字和字之间产生了因果关系。前面的上下文信息通过记忆缓存（KV Cache）逐层传递，后面的每个字都是基于前面所有字生成的，不再是独立随机采样。suffix decay问题由此得到抑制。

## 第二层：从固定批改到智能阅卷

草稿生成解决了猜得准的问题，验证层要解决验得省的问题。

DSpark在草稿模型的输出端加了一个极小的**打分器**（置信度预测头）。每生成一个候选字，打分器就输出一个0到1之间的分数——表示这个字被大模型认可的概率。这个打分器的参数量极小，推理延迟几乎可以忽略。

拿到分数分布后，**智能调度器**做两件事：

**第一，动态决定验多少字。** 如果一块草稿的分数整体偏高（比如前16个字信心分都在0.8以上），调度器就多验几个，让大模型一次多确认几个字。如果分数分布参差不齐，调度器就截断低分后缀，避免大模型浪费时间确认明显会错的字。

**第二，优先验高分的字。** 调度器不是死板地从左到右验，而是按分数从高到低排序。这样做的好处是：即使验到某个位置被大模型否定，前面高分的字已经被接受了，没有浪费。

还有一个容易被忽略的细节——**时序校准**。打分器在训练时有个毛病：对前面的字过于自信，对后面的字信心不足。DSpark通过一个温度参数动态校正这种偏差，让分数更接近真实的被接受概率。

这个设计的工程价值，在**高并发场景**下体现得最充分。几十个用户同时请求时，系统负载波动大，固定验证长度会造成严重的算力浪费。动态调度让验证资源跟着草稿质量走——好草稿多验，差草稿少验。

```
验证调度决策流程（单步）
├─ 输入：草稿 [字1, 字2, ..., 字γ] + 信心分 [s1, s2, ..., sγ]
├─ Step 1：时序校准
│   └─ 后面的字信心分适当上调，避免系统性低估
├─ Step 2：动态截断
│   └─ 找到第一个信心分低于阈值的位置
│   └─ 只验前面高分部分，后面的直接放弃
├─ Step 3：按分数排序验证
│   └─ 先验最有把握的字，再验不确定的字
└─ 输出：确认保留多少字 + 从哪开始重新生成
```

## 生产数据与DeepSpec开源

DSpark在DeepSeek-V4生产环境中的实测数据：

| 模型 | 基线 | per-user速度提升 |
|:---|:---|:---|
| DeepSeek-V4-Flash | 传统逐字生成 | **60%-85%** |
| DeepSeek-V4-Pro | 传统逐字生成 | **57%-78%** |

这不是实验室跑分，是生产环境真实用户请求的统计结果。V4-Flash的提升区间比V4-Pro更宽，因为轻量模型的草稿生成延迟更低，动态调度的收益空间更大。

比DSpark本身更值得关注的，是同步开源的**DeepSpec**代码库。MIT协议，全栈覆盖：

- **数据准备**：从你的大模型提取训练数据，构建草稿模型训练集
- **草稿训练**：支持DSpark、DFlash、Eagle3三种架构的端到端训练
- **引擎对接**：与vLLM、TGI等主流推理引擎的集成代码
- **效果评测**：接受率、速度提升、吞吐量的标准化测试

DeepSpec的**通用性**是关键。DSpark加速的是DeepSeek V4，但DeepSpec让你可以拿自己的模型——不管是Llama、Qwen还是自研模型——训练一个专属的草稿加速模块。

## 为什么这件事比发新模型更重要

DSpark的发布传递了一个信号：**大模型行业的竞争焦点，正在从谁的参数多转向谁的推理快**。

训练万亿参数模型需要几亿美元，只有头部公司玩得起。但推理优化是每家部署大模型的企业都面临的刚需——同样的GPU集群，推理速度快一倍，就意味着服务用户多一倍，或者成本降一半。

从工程角度看，DSpark的**可组合性**是最大亮点。它不改动V4-Pro和V4-Flash的基座权重，只是一个外挂模块。企业可以在不重新训练大模型的前提下，附加DSpark获得60%以上的推理加速。这种即插即用的优化方案，比推一个新版本模型的落地门槛低得多。

当然，DSpark也有局限。加速效果取决于草稿模型与大模型的默契程度——两者在词汇偏好上的重叠度越高，猜中率越高。如果大模型经过大量后训练微调，草稿模型可能需要同步更新。

但无论如何，DeepSeek用DSpark证明了一件事：**在推理侧，还有大量工程优化空间等待被挖掘。** 万亿参数的竞赛已经接近尾声，但推理效率的竞赛才刚刚开始。
